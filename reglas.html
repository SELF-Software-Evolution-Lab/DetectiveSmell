Reglas

En DetectiveSmell, se ha identificado ciertas reglas que se consideran fundamental para mantener la calidad del código en los tres frameworks admitidos: SpringBoot, Angular y NestJS. Estas reglas actúan como directrices esenciales para garantizar que el código desarrollado en estos entornos no se convierta en un 'code smell'. Al respetar estas reglas, los desarrolladores pueden mejorar la legibilidad, la escalabilidad y la mantenibilidad de su código, asegurando así un desarrollo más eficiente y una menor probabilidad de introducir defectos. Mantener la coherencia en la aplicación de estas reglas en todos los frameworks ayuda a establecer un estándar de calidad uniforme en todo el proyecto

SpringBoot NestJS Angular

Reglas SpringBoot

Todos los atributos de las entidades son objetos

Esta regla asegura que todas las entidades sean objetos para permitir una mayor flexibilidad y escalabilidad en el código. Evita el uso de tipos primitivos que pueden limitar la funcionalidad.

Ejemplo:

@Entity
public class Usuario {
    private Long id;
    private String nombre;
}

Error:

@Entity
public class Usuario {
    private long id;
    private String nombre;
}

Todas las entidades tienen la anotación @Data

La anotación @Data de Lombok genera automáticamente getters, setters, equals, hashCode y toString. Esto ayuda a reducir el boilerplate en las entidades.

Ejemplo:

@Data
@Entity
public class Usuario {
     private Long id;
     private String nombre;
}

Error:

@Entity
public class Usuario {
    private Long id;
    private String nombre;
}

Ver error

La anotación @Service se utiliza en la capa de servicio donde se escribe la lógica de negocio. Esto ayuda a la inyección de dependencias y a la transaccionalidad.

    ...

Todos los atributos de la clase son serializables

Esta regla asegura que todas las clases sean serializables para permitir la transferencia de objetos a través de la red o el almacenamiento en archivos. Esto es útil para la persistencia y la comunicación entre sistemas.

Ejemplo:

public class Usuario {
     private Long id;
     private String nombre;
}

Error:

public class Usuario {
     private long id;
     private string nombre;
}

Ver error

Todos los métodos de tipo GET deben retornar atributos de tipo DetailDTO

Esta regla asegura que los métodos de tipo GET retornen atributos de tipo DetailDTO para evitar la exposición de información sensible o redundante. Esto ayuda a mantener una arquitectura limpia y coherente.

Ejemplo:

@GetMapping("/usuarios/{id}")
public DetailDTO getUsuario(@PathVariable Long id) {
     Usuario usuario = usuarioService.findById(id);
     return new DetailDTO(usuario);
}

Error:

@GetMapping("/usuarios/{id}")
public DetailDTO getUsuario(@PathVariable Long id) {
     Usuario usuario = usuarioService.findById(id);
     return usuario;
}

Ver error

Reglas Angular

No debería haber servicios ni componentes a nivel de la carpeta 'app' que no sean los propios de 'app'

Se establece como buena práctica mantener la carpeta 'app' exclusivamente para los archivos que pertenecen directamente a la lógica y la interfaz de usuario de la aplicación. Siguiendo esta convención, se promueve una estructura organizada y mantenible que facilita la comprensión y el mantenimiento del código.

Ejemplo:

Carpeta: App
 Archivos: app.component.ts

Error:

Carpeta: App
 Archivo: user.component.ts

Ver error

Los nombres de las carpetas (módulos) deben coincidir con sus archivos

Se recomienda que los nombres de las carpetas que actúan como módulos coincidan exactamente con los nombres de sus archivos correspondientes. Esta práctica promueve la legibilidad, la comprensión y la coherencia en el código, facilitando la navegación y el mantenimiento del proyecto.

Ejemplo:

Carpeta: User
 Archivos: user.ts

Error:

Carpeta: User
 Archivos: client.ts

Ver error

Los nombres de los módulos deben estar en singular

Esta práctica asegura una coherencia uniforme en toda la estructura del proyecto, lo que simplifica la comprensión y la mantenibilidad del código. Al mantener los nombres de los módulos en singular, se evitan ambigüedades y se facilita la identificación clara de cada componente de la aplicación.

Ejemplo:

Modulo: clase.module.ts

Error:

Modulo: clases.module.ts
